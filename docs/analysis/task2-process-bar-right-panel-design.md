> 分析日期: 2026-04-23
> 触发原因: 用户反馈 — 倾向右侧栏位 Process Bar，承载推进阶段、Context、后续事项等丰富信息
> 分析目标: 评测右侧栏位设计的合理性，更新 Case Study 布局与信息架构

## 一、设计决策评测：右侧栏位 Process Bar

### 1.1 用户原始反馈解读

用户的核心诉求：
- **位置**: Process Bar 从"底部常驻"或"抽屉"改为**右侧固定栏位**
- **信息密度**: 不仅记录阶段，还需承载 **Context 信息 + 后续待办 + 推进上下文**
- **本质定位**: Process Bar 不是简单的"进度指示器"，而是**"推进上下文面板"（Progression Context Panel）**

### 1.2 评测结论：右侧栏位是非常合理的设计选择

| 维度 | 右侧栏位评价 | 分析 |
|------|-------------|------|
| **信息展示空间** | ✅ 优秀 | 垂直方向天然适合阶段时间线（Step 1 → Step 2 → Step 3 自上而下）。相比底部水平条的"空间饥饿"，右侧栏位提供了充裕的纵向空间来展开每个阶段的上下文 |
| **与主内容区关系** | ✅ 合理 | 右侧栏位与 Main Content 形成"主-辅"并列结构。用户的视觉焦点在左侧主内容，右侧上下文作为**常驻参考**，扫视即可获取全局状态，无需切换 |
| **阶段推进的可视化** | ✅ 优秀 | 纵向时间线比水平进度条更符合人类对"流程推进"的心智模型（从上到下 = 从开始到结束） |
| **上下文信息的承载** | ✅ 优秀 | 每个阶段节点下方可以展开/常驻显示：该阶段的上下文摘要、待办子项、AI 建议、关联文档等。这是底部进度条完全无法承载的 |
| **屏幕空间占用** | ⚠️ 需注意 | 右侧栏位会压缩 Main Content 的可用宽度。建议设计为**可折叠/可调整宽度**（如 VS Code 的 Sidebar），用户可根据需要展开或收起 |
| **移动端适配** | ⚠️ 需设计 | 移动端右侧栏位可以转化为**底部 Sheet** 或**可滑出的 Drawer**，保持信息架构一致，仅改变呈现形式 |

### 1.3 与之前方案的对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **抽屉形态**（Task 1 原始） | 节省空间 | 隐藏关键信息，用户需主动打开 | 不适用于需要频繁查看上下文的场景 ❌ |
| **底部常驻**（Task 2 初版建议） | 常驻可见 | 水平空间严重不足，只能显示极简进度 | 仅适合"纯进度指示"，无法承载上下文 ❌ |
| **右侧栏位**（用户反馈） | 常驻 + 充裕空间 + 纵向时间线 | 压缩主内容宽度 | **推进上下文面板**的最优解 ✅ |

**结论**: 用户提出的右侧栏位方案是三者中的最优解，应采纳。

### 2.2 右侧栏位信息分层

右侧栏位 Process Bar 采用**三层信息架构**：

```
┌──────────────────────────────┐
│  Layer 1: 流程概览            │  ← 常驻显示
│  ─────────────────────────   │
│  流程标题 + 整体进度环        │
│  [●──●──▶──○──○] 40%       │
├──────────────────────────────┤
│  Layer 2: 阶段时间线          │  ← 常驻显示
│  ─────────────────────────   │
│  ● 选题 ✓                    │
│  ● 大纲 ✓                    │
│  ▶ 撰写 ◀── 当前             │  ← 高亮展开
│  │   ├─ 上下文摘要            │
│  │   ├─ 待办子项              │
│  │   └─ AI 洞察               │
│  ○ 审核                      │
│  ○ 发布                      │
├──────────────────────────────┤
│  Layer 3: 操作与导航          │  ← 常驻显示
│  ─────────────────────────   │
│  [💡 AI 建议] [📋 流程图]    │
│  [⬅ 上一步] [下一步 ➡]      │
└──────────────────────────────┘
```

### 2.3 各层信息详细设计

#### Layer 1: 流程概览（Flow Overview）

| 元素 | 说明 | 示例 |
|------|------|------|
| 流程标题 | 当前执行的 Workflow 名称 | "内容发布: React Server Components" |
| 进度环 | 圆形或线性进度，显示完成百分比 | 40%（2/5 步骤完成） |
| 时间信息 | 已用时间 / 预计剩余时间 | "已进行 45 分钟 · 预计还需 1.5 小时" |
| 保存状态 | 最近一次 Draft 保存时间 | "自动保存: 3 分钟前" |

#### Layer 2: 阶段时间线（Stage Timeline）⭐ 核心

每个阶段节点的显示模式：

**已完成阶段**（●）:
```
● 选题 ✓
  └─ 完成于: 今天 10:15
  └─ 耗时: 15 分钟
```

**当前阶段**（▶）— 展开显示详细信息:
```
▶ 撰写 ◀── 当前步骤
  ├─ 上下文摘要
  │   "正在撰写技术博客正文，
  │    已完成 2/6 节，当前编辑:
  │    第 3 节 '最佳实践'"
  │
  ├─ 待办子项
  │   ☐ 补充代码示例（3 处）
  │   ☑ 添加 RSC vs SSR 对比表
  │   ☐ 插入性能基准数据
  │
  └─ AI 洞察
      💡 "检测到'最佳实践'章节，
         建议参考 Next.js 官方文档
         的 3 条最新推荐"
      [查看详情] [应用建议]
```

**待开始阶段**（○）:
```
○ 审核
  └─ 分配给: 老张
  └─ 预计开始: 撰写完成后自动通知
```

#### Layer 3: 操作与导航（Actions & Navigation）

| 按钮 | 功能 | 场景 |
|------|------|------|
| 💡 AI 建议 | 触发 AI 对当前 Context 的分析 | 用户卡壳时寻求建议 |
| 📋 流程图 | 展开全屏流程图视图 | 需要全局视角时 |
| ⬅ 上一步 | 回退到上一个阶段（带确认） | 发现前面步骤有误 |
| 下一步 ➡ | 推进到下一个阶段（需验证） | 当前阶段完成 |
| 📌 固定此栏 | 锁定右侧栏位宽度 | 防止误操作收起 |

## 四、右侧栏位 Process Bar 的关键设计原则

### 4.1 信息密度控制

右侧栏位信息丰富是一把双刃剑，需遵循以下原则：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **当前步骤优先** | 只有当前步骤展开显示详细上下文，其他步骤折叠 | 点击步骤节点可手动展开/折叠 |
| **三层信息分层** | Layer 1（概览）常驻，Layer 2（时间线）常驻，Layer 3（操作）常驻 | 视觉层级通过字体大小、颜色、间距区分 |
| **智能折叠** | 当阶段数量 > 7 时，已完成阶段自动折叠为"..." | 点击展开查看完整历史 |
| **上下文摘要限制** | 每个阶段的上下文摘要不超过 3 行 | 超出时显示"..." + [展开] 按钮 |

## 五、总结

### 5.1 对右侧栏位 Process Bar 的最终评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 信息承载能力 | ⭐⭐⭐⭐⭐ | 纵向空间充裕，可展示阶段、上下文、待办、AI 洞察等多维信息 |
| 视觉可扫描性 | ⭐⭐⭐⭐⭐ | 常驻显示，用户扫视即可获取全局状态，无需交互 |
| 与主内容区协作 | ⭐⭐⭐⭐ | 主-辅并列结构合理，但需注意宽度压缩问题（可通过可调整宽度解决） |
| 移动端适配 | ⭐⭐⭐ | 需额外设计底部 Sheet 形态，增加开发成本 |
| 跨角色协作 | ⭐⭐⭐⭐⭐ | 不同角色看到的右侧栏位内容不同（小明看撰写上下文，老张看审核清单），天然支持角色定制 |

**总体评价**: 右侧栏位 Process Bar 是**"推进上下文面板"的最优载体**，充分满足用户对信息丰富度和常驻可见性的需求。建议采纳为默认设计，并配套实现可调整宽度和移动端底部 Sheet 适配。

### 5.2 与 Task 2 初版分析的对比更新

| Task 2 初版建议 | 用户反馈后更新 | 理由 |
|----------------|---------------|------|
| Process Bar 底部常驻 | ❌ 废弃 | 水平空间不足，无法承载上下文信息 |
| Process Bar 抽屉形态 | ❌ 废弃 | 隐藏关键信息，与"推进上下文"定位矛盾 |
| **Process Bar 右侧栏位** | ✅ **采纳** | 纵向空间充裕，常驻可见，支持丰富信息分层 |
| 左右布局（Main Content） | ✅ 保留 | 与右侧栏位形成"主-辅"双栏，信息架构更清晰 |

### 5.3 后续建议

1. **原型验证**: 建议先用 Figma/Whimsical 制作右侧栏位 Process Bar 的交互原型，验证信息密度是否可接受
2. **用户测试**: 邀请 3-5 名目标用户测试"撰写阶段"的右侧栏位体验，特别关注：信息是否过多、待办清单是否实用、AI 洞察是否干扰
3. **性能考量**: 右侧栏位信息实时更新（如字数统计、AI 洞察推送）需要高效的状态同步机制，避免主内容区卡顿
