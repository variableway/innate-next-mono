# Task 1: Intent Driven 产品开发流程分析

> 分析日期: 2026-04-23
> 分析对象: tasks/product/general.md — Task 1 (Intent Driven 方式)
> 分析维度: 产品设计合理性、技术可行性、用户体验、开发流程

---

## 一、核心观点梳理

Intent Driven 产品开发流程的核心主张：

### 1. 产品运行态 — 以 Use Case 驱动的交互范式

| 组件 | 职责 | 关键特性 |
|------|------|---------|
| **Main Content Page** | 主线内容区 | 展示总体步骤、当前操作步骤、支持页面直接操作 |
| **Process Bar** | 进度与导航 | 抽屉形态，显示当前步骤位置和整体进度 |
| **Command Box** | 意图输入 | 输入自然语言/结构化指令，调用 Skill Set 执行操作 |
| **Skill Set** | 能力层 | 可自定义或复用第三方，覆盖 API 调用、页面操作、UI 交互等 |

核心交互规则：**Command Box 指令操作** 与 **Main Content 可视化操作** 双向等价，均可执行 CRUD，状态实时同步至 Process Bar。

### 2. 产品开发态 — 从需求到 Spec 的流水线

```
Use Case 分析 → 流程图
       ↓
Domain 概念设计 + UI CRUD Page 设计
       ↓
Domain 分割 → Domain Spec 文件
       ↓
最终交付物：Use Case 说明 / Domain Spec / UI Spec
```

UI CRUD Page 作为 Sidebar Submenu，与主流程解耦，可随时调用。

## 二、Intent Driven 分析

### 2.1 非常合理的部分 ✅

| 观点 | 合理性分析 |
|------|-----------|
| **从完整 Use Case 出发** | 符合现代产品设计第一性原理。以用户场景而非功能清单为起点，确保每一步开发都锚定真实价值，避免功能蔓延 |
| **Command Box 作为意图入口** | Intent Driven 的核心载体。用户表达"做什么"而非"怎么做"，大幅降低复杂系统的认知负荷。特别适合 AI 时代的人机协作范式 |
| **Skill Set 可插拔架构** | 将业务能力抽象为可复用、可组合的单元，符合模块化设计原则。自定义 + 第三方复用的双轨制兼顾灵活性与生态扩展 |
| **可视化操作与指令双向等价** | 关键设计。避免"指令优先"排斥非技术用户，也避免"纯 GUI"限制效率型用户。双轨制是 Intent Driven 落地的最优解 |
| **主流程与 CRUD 分离** | 高度合理。主流程聚焦"推进任务"，CRUD 聚焦"管理数据"，两者认知模型不同。Sidebar Submenu 的解耦方式既保证主线清晰，又确保数据操作随时可达 |
| **DDD 先 Domain 后 UI** | 符合领域驱动设计（DDD）的正向流程。先建立统一语言（Ubiquitous Language），再映射到界面，避免"界面驱动反模式" |
| **Spec 文件作为交付物** | 合理。形成可沉淀、可版本化、可被 AI 理解的结构化知识资产，为后续自动化生成代码奠定基础 |
| **Process Bar 进度可视化** | 长流程必须提供进度感知和上下文定位，符合 Nielsen 可用性启发式原则（系统状态可见性） |

### 2.2 非常不合理的部分 ❌

| 观点/现状 | 不合理性分析 |
|-----------|-------------|
| **"按固定流程生成步骤"** | 与 **Intent Driven** 的本质存在深层矛盾。Intent Driven 的核心是"用户意图驱动动态流程分叉"，而"固定流程"意味着预定义路径、强制步骤顺序。如果流程是固定的，Command Box 就只是"快捷操作栏"而非"意图引擎" |
| **"每个步骤不强制完成，可以同时做"** | 过度理想化。真实业务流充满依赖关系（如：下单前必须登录、支付前必须确认地址）。无约束的并行会导致状态冲突、数据不一致、业务规则被破坏。必须有显式的**依赖图（DAG）**定义哪些可并行、哪些必须串行 |
| **Process Bar 采用"抽屉形状"** | 进度信息属于**高频查看、低操作**的上下文，抽屉的"收起-展开"交互会隐藏关键状态。Process Bar 应常驻可见（如侧边固定栏或顶部时间轴），抽屉仅用于展示步骤详情 |
| **"CRUD 操作反应在 Process Bar"** | 概念层级混淆。Process Bar 应反映**流程阶段推进**（如：申请 → 审核 → 批准），而非**数据变更**（如：修改用户名字段）。数据 CRUD 属于状态变化，流程推进属于阶段跃迁。强行映射会导致进度条语义失真 |
| **缺少错误处理与回退机制** | 长流程必须支持：步骤回退、指令撤销、异常分支。当前设计完全未提及，这是从 Demo 到生产的关键差距 |
| **Command Box 无发现机制** | 用户如何知道有哪些 Skill 可用？如何知道当前上下文支持什么指令？没有 Auto-complete、Help、Skill Discovery 机制，Command Box 会很快沦为"猜命令游戏" |

### 2.3 需要优化的部分 ⚠️

| 现状 | 优化建议 |
|------|---------|
| **Command Box 输入形式未定义** | 需明确支持三种输入模式的层级：\n1. **自然语言**（"帮我创建一个订单"）→ 高灵活性、低精确度\n2. **结构化指令**（"/order create --product=x"）→ 中灵活性、高精确度\n3. **快捷命令**（"/order" 呼出表单）→ GUI 与 CLI 的混合态 |
| **Skill 的上下文感知未说明** | Skill 是否需要感知当前流程阶段、用户角色、页面状态？例如：在"审核中"阶段调用"修改订单"Skill 是否被允许？ |
| **流程状态持久化策略缺失** | 用户刷新页面、切换设备、会话超时后，流程状态如何恢复？需要服务端状态机 + 本地缓存的双层持久化 |
| **Domain Spec 与 UI Spec 的生成工具链未定义** | 是从流程图手动推导？还是 AI 辅助生成？如果依赖 AI，Prompt 工程和质量控制策略是什么？ |
| **Use Case 到流程图的粒度未明确** | 一个 Use Case 对应一个流程还是多个子流程？流程图是 BPMN 标准还是自定义格式？这直接影响后续 Domain 分割的边界 |
| **UI CRUD Page 与主流程的数据联动** | 当用户在 Sidebar 的 User CRUD Page 修改了当前登录用户的信息，Main Content Page 是否需要实时刷新？需要定义全局状态管理和事件总线策略 |
| **权限模型完全缺失** | 谁可以执行什么指令？哪些 Skill 需要审批？流程中的哪些步骤可以跳过？必须引入 RBAC 或 ABAC 模型 |

### 2.4 可以放弃的部分 🗑️

| 内容 | 放弃理由 |
|------|---------|
| **"固定流程"的绝对化表述** | 应改为**"流程模板 + 动态分支"**。提供默认路径，但允许用户意图触发分叉、跳过、回退。固定流程在 Intent Driven 范式下是反模式 |
| **Process Bar 作为唯一进度表示** | 建议增加**迷你进度指示器**（如面包屑、步骤圆点）常驻在页面顶部，抽屉仅用于查看完整步骤详情和历史记录 |
| **Command Box 作为"高级功能"的隐含定位** | 不应将 Command Box 视为 GUI 的增强或替代，而应作为**平行的第一入口**。甚至可以考虑将 Command Box 设为默认焦点，引导用户习惯意图驱动的工作流 |
| **完全独立的 UI CRUD Page** | 若 CRUD 操作与主流程数据高度相关，应考虑**内联编辑**（Inline Edit）或**模态框**作为首选，Sidebar Submenu 仅用于"全局数据管理"类 CRUD |

### 2.5 如何实现、难点与风险

#### 推荐实现路径

```
Phase 1: 核心抽象层（4-6 周）
  ├─ 定义 Intent 解析协议（自然语言 → 结构化 Intent JSON）
  ├─ 设计 Skill Registry（注册、发现、版本、权限）
  └─ 构建最小流程引擎（串行步骤 + 基础状态机）

Phase 2: 交互层（3-4 周）
  ├─ 双模 UI（Command Box + Main Content 实时同步）
  ├─ Process Bar 常驻组件 + 抽屉详情
  └─ Sidebar CRUD Page 框架

Phase 3: 流程智能化（4-6 周）
  ├─ 动态分支引擎（条件跳转、用户意图路由）
  ├─ 并行步骤协调器（DAG 执行 + 冲突检测）
  └─ 撤销/重做系统（操作日志 + 补偿事务）

Phase 4: 开发工具链（3-4 周）
  ├─ Use Case → 流程图 → Domain Spec 的半自动生成
  ├─ Spec 驱动的 UI 代码生成
  └─ Skill SDK（让第三方开发者可以贡献 Skill）
```

#### 核心技术难点

1. **意图解析的精确性与召回率平衡**
   - 纯自然语言灵活性高但歧义大，纯结构化命令精确但学习成本高
   - **解法**：分层解析 — 先匹配结构化命令模式，fallback 到 LLM 语义解析，最终人工确认兜底

2. **流程状态机的正确性保证**
   - 支持分支、并行、回退后，状态空间爆炸，极易出现死锁或状态不一致
   - **解法**：采用 Petri Net 或有限状态机（FSM）的严格数学模型，配合形式化验证工具

3. **双模操作的实时同步**
   - 用户在 Command Box 输入"/next" 和 点击 Main Content 的"下一步"按钮必须产生完全一致的状态变更
   - **解法**：所有操作统一收敛到 **Action/Event 层**，GUI 和 CLI 只是不同触发器，不直接操作状态

4. **Skill 的上下文隔离与共享**
   - Skill A 修改了全局状态，Skill B 的后续执行如何感知？
   - **解法**：定义显式的 **Context Bag**（上下文背包），Skill 读取/写入需声明，运行时注入

#### 主要风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| **Intent Driven 对用户学习曲线陡峭** | 🔴 高 | 提供"新手模式"（强 GUI 引导）→ "进阶模式"（混合使用）→ "专家模式"（ predominantly CLI）的渐进式暴露 |
| **流程引擎复杂度导致项目延期** | 🔴 高 | Phase 1 严格限制为"串行步骤 + 基础状态"，拒绝早期引入并行和分支。用 MVP 验证核心价值后再扩展 |
| **LLM 依赖带来的成本与延迟** | 🟡 中 | 意图解析优先使用本地规则引擎（如 regex + 模板匹配），LLM 仅用于 fallback 和复杂语义理解 |
| **双模同步的边际 bug** | 🟡 中 | 建立自动化测试矩阵：所有操作必须通过 GUI 和 CLI 两条路径各执行一次，断言最终状态一致 |
| **Skill 生态冷启动** | 🟡 中 | 初期提供 10-15 个高价值内置 Skill 覆盖核心场景，降低用户"无 Skill 可用"的挫败感 |
| **CRUD 与主流程的数据竞态** | 🟡 中 | 引入乐观锁或最终一致性策略，冲突时以流程引擎状态为准，CRUD 操作触发刷新提示 |

## 三、总结

### 总体评价

Intent Driven 的产品开发流程**方向具有前瞻性和战略价值**。在 AI 重塑人机交互的转折点，"意图优先、双模并行、流程可视化"的设计范式是合理的下一代产品形态。

但当前方案存在**概念层级的混淆**：
- "固定流程"与"Intent Driven"的本质冲突需要调和
- "Process Bar 映射 CRUD"的语义错位需要修正
- 生产级必需的权限、错误处理、状态持久化等机制完全缺失

### 关键行动项（按优先级）

1. **【紧急】重新定义流程模型** — 放弃"固定流程"表述，改为"流程模板 + 意图分叉"的动态模型
2. **【高】分离"流程阶段"与"数据状态"** — Process Bar 只反映阶段推进，CRUD 状态变更通过单独的通知/日志机制呈现
3. **【高】设计 Skill Discovery 机制** — Command Box 必须提供 Auto-complete、Help、Context-aware Suggestion
4. **【中】定义意图解析分层策略** — 结构化命令（精确）+ 自然语言（灵活）+ 人工确认（兜底）
5. **【中】补全生产级机制** — 权限模型（RBAC）、撤销/重做、状态持久化、错误分支
6. **【低】优化 Process Bar 形态** — 常驻迷你指示器 + 抽屉详情视图，而非纯抽屉
